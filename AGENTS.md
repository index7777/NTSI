# NTSI 專案強制開發規範

## 開始工作前

每次接手、同步、模型切換或上下文壓縮後，先讀取 docs/production/HANDOFF.md 與 docs/production/WORK_QUEUE.md，依工作主檔最早未驗收項目恢復進度。使用者對話中的修改、錯誤回報與指定作法是最新要求，不得以舊文件或舊程式覆蓋。

任何涉及人物、NPC、玩家外觀、服裝、劇情、對話演出、UI 人像或生成美術資產的任務，在分析或動工前必須完整閱讀：

1. `docs/art/CHARACTER_PORTRAIT_STANDARD.md`
2. `docs/art/角色美術與境界演出母板.md`
3. `art/README.md`
4. `game/src/config/characters.ts`
5. `docs/art/UI_STYLE_GUIDE.md`

本步驟不可略過；若規格與使用者最新指示衝突，以使用者最新指示為準，並同步更新規格。

## 人物與對話

- 場景需要人物站位或動作時使用完整立繪；不需要完整身形時使用符合母板的胸像頭像。
- 禁止以圓形、三角形、矩形或其他程式幾何人偶代替已有角色。
- 角色名稱不得使用【】包圍；系統提示與角色對話必須使用不同 HUD 樣式。
- 小遊戲選擇卡只顯示玩法名稱與必要狀態，不得顯示「點擊開始」等多餘操作文字；互動性以游標、觸控回饋與動效表達。
- 玩家男女角、既有 NPC 與所有新角色都必須納入角色資產清單。
- 角色更換服裝、身份或境界時，必須建立獨立服裝版本及該版本所需表情頭像，不得覆蓋舊版本。

## 生成與新增美術

- 所有新美術任務必須先讀取 `assets-spec/style-guide.md` 與對應的 `assets-spec/*.yaml`；沒有規格項目時先建立規格，不得直接自由組 Prompt。
- 圖片生成一律使用 Codex 內建圖片工具；本專案不得建立或呼叫 OpenAI Image API 生成腳本。
- 生成 Prompt 必須先由 `python tools/compose_prompt.py --spec <spec-file> --id <asset-id>` 組裝，再補入使用者本次明確要求；不得刪除或忽略 YAML 的 reference、identity lock、constraints 與 manual checks。
- 原始生成結果進 `art/raw`，可供比較的版本進 `art/candidates`；禁止將 raw 或 candidates 直接放入 `game/public/assets`。
- 每項最多進行三輪生成修正；三輪仍不符時標記阻塞原因，不得無限制盲目重試。
- 每組候選必須以 `tools/make_contact_sheet.py` 建立 Contact Sheet，供人工驗收。
- 備選圖只有在使用者確定採用後才可去背；去背使用既有離線工具 `tools/去背工具.html`，結果存入 `art/processed`。
- processed 資產必須以 `tools/validate_asset.py` 驗證；程式驗證只涵蓋尺寸、Alpha、裁切與色鍵殘留，人物身份、手部、透視、光影、文字與幾何仍需人工驗收。
- 通過程式驗證且取得使用者定案後，才可移入 `art/active` 並建立 `game/public/assets` 執行副本；不採用版本移入 `art/rejected`，被取代正式版本移入 `art/archive`。
- Sprite Sheet 只能由已驗證影格建立；呼吸、漂浮、搖擺、霧、雲與一般 UI 回饋優先使用 Phaser Tween、圖層切換、粒子或 Shader，不得要求 AI 自由生成傳統逐幀動畫。
- runtime 資產新增或改名後必須執行 `npm run manifest:assets` 與 `npm run validate:assets`。
- 新人物必須先建立完整角色母板、中性立繪、同框比例驗收與 neutral 頭像，再製作其他表情。
- 新表情只可調整眉、眼、嘴及合理的細微神態；臉型、五官位置、髮型、服裝、鏡頭與裁切必須一致。
- 圖片內不得生成遊戲文字；繁體中文由遊戲 UI 即時渲染。
- 原始圖放入 `art/` 對應分類；遊戲執行副本放入 `game/public/assets/`。
- 新增、改名或搬移資產時，必須同步更新角色清單、交接文件與資產搬移對照。
- 活動劇情場景卡必須依玩家性別與當前境界使用已驗收服裝版本；不同境界可升級卡片演出，但不得自行改變小遊戲操作規則。
- 所有遊戲場景、場景卡與 UI 母板一律以 16:9 橫式設計；舊 9:16 圖只能作為內容參考，不得裁切、拉伸或宣稱為正式橫式素材。
- 首圖與核心常駐場景必須規劃卯時、午時、酉時三個同鏡頭版本；小遊戲在玩法與基準美術定案前不得提前生成午時、酉時版本。
- UI 與切換流程母板只能作為實機配置、UI、UX、HUD 及互動參考，禁止整張放入遊戲或裁成一張假介面。
- 禁止使用圖片生成取代 HUD 圖層、按鈕、進度條、功能圖示、文字、互動狀態、頁面切換或程式功能。
- UI 必須拆成可獨立維護的場景、人物、面板、框線、圖示、按鈕狀態、進度條與文字圖層；互動與數值由程式實作。
- 小遊戲基礎時限為 30 秒並必須顯示倒數；時間到未達成功條件一律標記失敗。
- 母板中的「宗門事務」與「散步遊歷」不是實機功能，禁止列入開發或資產製作。

## 完成條件

- 未通過母板、透明背景、同框比例、表情一致性及手機顯示驗收的素材，只能標記為候選，不得宣稱正式完成。
- 修改程式後必須執行 `npm run validate:balance`、`npm run validate:landscape`、`npm run validate:assets` 與 `npm run build`。

## 工作進度溝通（所有接手對話強制遵守）

- 禁止在顯示圖片生成結果、下載／授權視窗、工具呼叫或任何中間產物後無說明停住；中間產物不是任務完成。
- 生成、下載、建置、測試、解壓或其他等待超過數秒的工作，開始前必須說明「正在做什麼、為什麼需要等待、完成後接哪一步」。
- 工具仍在執行時必須明確說明目前正在等待工具結果；若不需要使用者操作，應說明「不是在等使用者」。
- 工具失敗、逾時、權限受阻或素材不符合時，必須立即說明具體原因、已完成部分、尚缺條件與採取的下一個替代方案。
- 每個工具步驟完成後應主動接續下一個安全且在範圍內的工作，不得把生成器彈窗、圖片預覽、命令輸出或無文字畫面當作回覆終點。
- 若因必要的使用者選擇、付費授權、缺少原始素材或其他新權限而確實無法繼續，必須清楚指出阻塞點並提出需要的決定，不得沉默等待。
- 專案總目標未完成時，完成單一頁面、單一素材、單一功能、一次建置、一次測試或一次階段驗收，都只算中間檢查點，不構成停止或結束回合的理由；回報該檢查點後必須立即進入玩家流程中的下一項工作。
- 「向使用者回報目前做到哪裡」不等於獲准暫停。除非使用者明確要求暫停，或出現無法在既有權限與專案資料內排除的實質阻塞，任何模型都不得在進度回報、成果截圖、測試通過或階段交付後自行結束工作。
- 本專案目前的持續工作終點是：完成2026-07-22與其後交辦素材的製作、驗收、接入，並依玩家流程完成至筑基九層圓滿；雷劫依使用者指示暫緩。在此終點前，只要仍有安全且可執行的下一步，就必須繼續。
- 全案後續開發只做原生1200×675橫式版本；舊540×960直式頁與直式素材只可作劇情／美術參考，不得裁切、縮放或維護成 runtime 備案，不得以「先相容直式」重複開發。新場景、UI、熱區、角色站位與小遊戲一律直接使用橫式座標。
- 正式美術特效必須以 Codex 可完成的最高品質製作：優先使用授權清楚的商用粒子／光效素材或生成專用透明特效層，再由 Phaser 以遮罩、混合模式、粒子系統、時序與鏡頭演出組合。CSS、簡單透明度動畫、幾何 Graphics 圓圈／線條或少量基本粒子只能作 debug，不得當成交付、驗收或完成依據。
- 合法停止條件僅限：使用者明確要求停止／暫停；必須由使用者做出會實質改變成果的選擇；缺少不可替代的原始素材、付費授權或新權限；工具或環境發生已嘗試替代方案仍無法排除的阻塞。停止時必須立即說明「為什麼停、正在等什麼、已嘗試什麼、解除後接哪一步」。
- 明確反例：完成首圖功能列、建置與4174驗收後，僅回報結果便結束回合，仍屬違規停工；正確行為是回報後直接接續問仙石、執事互動、雜役處與後續玩家流程。

## 本機雙埠測試

- `127.0.0.1:4173` 保留給使用者持續觀看開發畫面；自動化測試不得占用、終止或重啟此埠的程序。
- Codex／自動化實機驗證預設使用 `127.0.0.1:4174`。
- 若4174已被占用，依序改用4175～4178；必須回報實際使用埠號，不得為搶埠而終止不明程序。
- 測試完成時只可終止自己明確啟動且已核對命令列屬於本專案測試埠的程序，不得影響使用者的4173預覽。
